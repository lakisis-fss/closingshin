# 04. 뉴스 수집기 하이브리드 워크플로우 (Hybrid News Collector)

이 문서는 04_collect_news.py에 적용된 네이버 API + 웹 크롤링 하이브리드 수집 방식의 동작 원리와 관리 방법을 설명합니다.

---

## 1. 개요 (Overview)

기존 네이버 오픈 API는 한 번의 검색에서 최대 1000개의 뉴스 데이터만 제공합니다. 하지만 삼성전자나 한화에어로스페이스와 같은 인기 종목은 하루에도 수백 건의 뉴스가 나오기 때문에, 1000건을 다 수집해도 불과 1, 2일 전 데이터까지만 도달하게 되어 과거 특정 날짜(예: 5~7일 전)의 수집이 불가능한 데이터 공백 문제가 있었습니다.

이 문제를 해결하기 위해 API의 안정성과 웹 검색의 정밀성을 결합한 하이브리드 수집 로직을 도입했습니다.

---

## 2. 동작 프로세스 (Process Flow)

주요 수집 로직은 다음과 같은 단계로 진행됩니다.

### Phase 1: 네이버 API 수집 (Standard)
1.  API 호출: 타겟 종목에 대해 날짜순(sort=date)으로 최신 뉴스부터 수집을 시작합니다.
2.  한계 체크: 최대 1000건(10페이지)까지 수집하며, 수집 도중 타겟 날짜(과거 7일 범위)를 모두 확보하면 즉시 종료합니다.
3.  트리거 감지: 만약 1000건을 다 수집했는데도 타겟 날짜에 아직 도달하지 못했다면(뉴스 양이 너무 많음), 자동으로 웹 크롤링 모드로 전환합니다.

### Phase 2: 웹 크롤링 전환 (Fallback Scraper)
1.  날짜 필터 URL 생성: 네이버 통합 검색 주소에 시작일(ds)과 종료일(de)을 동일하게 설정하여 호출합니다.
    - 예: &ds=2026.03.11&de=2026.03.11
2.  Fender UI 분석: 네이버의 최신 검색 UI(Fender UI) 구조를 분석하여 data-heatmap-target 속성을 기반으로 제목, 본문, 언론사를 안정적으로 추출합니다.
3.  데이터 병합: API 데이터와 크롤링 데이터를 중복 검사하여 하나의 리스트로 통합합니다.

---

## 3. 보안 및 안정성 (Security & Anti-Bot)

네이버의 자동화 수집 차단을 방지하기 위해 다음 전략이 적용되었습니다.

- User-Agent 순환 (UA Rotation): 다양한 최신 브라우저 정보를 번갈아 사용하여 실제 사람이 접속하는 것처럼 위장합니다.
- 랜덤 지연 (Random Delay): 크롤링 모드 진입 시 2.0초에서 4.0초 사이의 불규칙한 대기 시간을 두어 기계적인 패턴을 숨깁니다.
- 조건부 실행 (Lazy Scraping): API만으로 데이터가 충분한 대다수의 종목(평균 90% 이상)은 크롤링을 수행하지 않으므로, 전체 시스템의 차단 위험을 최소화합니다.

---

## 4. 데이터 정제 로직 (Data Normalization)

수집된 뉴스는 다음 기준에 따라 가치가 평가됩니다.
- 언론사 신뢰도 가중치: 한경, 매경, 연합뉴스 등 주요 경제지 및 통신사 기사에는 높은 기본 점수(0.85 ~ 0.9)를 부여합니다.
- 유사도 기반 중복 제거: difflib 라이브러리를 통해 제목이 60% 이상 비슷할 경우 동일 기사로 간주하여 하나만 남깁니다.

---

> 관리 참고 사항
> 만약 특정 날짜의 수급/심리 점수가 0점이 나온다면, PocketBase `news_reports` 컬렉션에서 해당 날짜의 레코드가 존재하는지, 하이브리드 로직이 정상적으로 발동했는지 확인해 주세요.
