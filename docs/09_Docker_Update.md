# 07. 시놀로지 NAS 도커 이미지 업데이트 가이드 (Docker Update Guide)


## 0 단계 시놀로지의 작업스케줄러나 ssh 터미널로 직접 명령어 입력

```
cd /volume1/docker/closingshin
docker-compose pull   # 무조건 레지스트리에서 최신 덮어씌우기
docker-compose up -d  # 변경된 점 확인하고 예전 교체 후 새 컨테이너 백그라운드 구동
docker image prune -f # 쓸모 없어진 옛날 찌꺼기 이미지 비우기
```

이 문서는 코드가 수정되거나 앱이 업데이트되었을 때, 변경된 내용을 시놀로지 NAS의 도커(Docker) 환경에 새롭게 반영(업데이트)하는 단계별 가이드입니다.
shin

## 1단계: 내 PC에서 새로운 도커 이미지 빌드하기

현재 열려있는 터미널(VS Code 등)에서 `ClosingSHIN` 프로젝트 최상단 폴더 위치로 이동한 뒤, 아래 명령어를 차례대로 실행합니다.

### 1-1. 이미지 빌드하기
방금 수정한 코드를 포함하여 새로운 이미지를 만듭니다. 명령어 끝의 마침표(`.`)를 꼭 입력해야 합니다.

```bash
docker build -t closingshin:latest .
# --no-cache 옵션은 캐시를 사용하지 않고 강제로 새로 빌드하라는 뜻입니다.
```
> [!NOTE] 참고사항
> 빌드가 완료될 때까지 약간의 시간(보통 1~3분)이 소요될 수 있습니다.

### 1-2. 이미지를 파일(.tar)로 압축해서 뽑아내기
NAS로 쉽게 옮기기 위해 방금 만들어진 이미지를 단일 파일로 묶어냅니다.

```bash
docker save closingshin:latest -o closingshin.tar
```
> [!NOTE] 참고사항
> 이 명령어가 끝나면 프로젝트 폴더 안에 `closingshin.tar` 파일(수백 MB ~ 1GB 이상)이 생성됩니다.

---

## 2단계: 생성된 파일을 NAS로 복사하기

내 PC에서 완성된 이미지 파일을 시놀로지 NAS로 옮깁니다.

1. 윈도우 탐색기나 시놀로지 NAS의 파일 관리자(File Station)를 엽니다.
2. 1단계에서 만들어진 `closingshin.tar` 파일을 NAS의 `docker/closingshin/` 폴더 안으로 덮어쓰기 복사합니다.

---

## 3단계: 시놀로지 NAS에서 기존 컨테이너 교체하기

이제 시놀로지의 **Container Manager** 애플리케이션을 열어서 새 이미지를 적용할 차례입니다.

### 3-1. 기존 프로젝트(컨테이너) 중지
1. Container Manager의 **[프로젝트]** 탭으로 이동합니다.
2. 현재 실행 중인 `closingshin` 프로젝트를 우클릭(또는 작업 버튼)하여 **[중지]**를 누릅니다.

### 3-2. 기존 이미지 삭제 (충돌 방지)
1. **[이미지]** 탭으로 이동합니다.
2. 목록에 있는 `closingshin:latest` 이미지를 찾아 우클릭 후 **[삭제]**합니다.
> [!WARNING] 삭제 오류 발생 시
> "사용 중이라 삭제할 수 없습니다"라는 오류가 뜬다면, 컨테이너 탭에서 해당 컨테이너가 확실히 중지 또는 삭제되었는지 확인하세요. 프로젝트를 재빌드할 것이기 때문에 컨테이너를 지워도 실제 데이터(JSON 파일 등)는 매핑된 폴더에 안전하게 남아있습니다.

### 3-3. 새로운 이미지 가져오기
1. 여전히 **[이미지]** 탭에서 상단의 **[가져오기]** 버튼을 누릅니다.
2. `파일에서 추가`를 선택하고, 아까 NAS에 복사해 둔 `docker/closingshin/closingshin.tar` 파일을 찾아 선택합니다.
3. 가져오기 작업이 끝나면 새로운 `closingshin:latest` 이미지가 목록에 다시 나타납니다.

### 3-4. 프로젝트 다시 시작 (새 이미지 적용)
1. 다시 **[프로젝트]** 탭으로 일동합니다.
2. 중지되어 있는 `closingshin` 프로젝트를 우클릭하여 **[빌드]**(또는 **[동작] -> [빌드]**)를 누릅니다.
3. 빌드 로그가 성공적으로 끝나면 창을 닫고 프로젝트를 **[시작]** 버튼을 눌러 실행합니다.

## 4단계: 역방향 프록시 + Docker 포트 설정 (HTTPS 외부 접속)

외부에서 `https://drspike.synology.me:4443`으로 접속하려면, 공유기 -> 시놀로지 DSM 역방향 프록시 -> Docker 컨테이너까지 트래픽이 올바르게 흘러야 합니다. 전체 흐름은 다음과 같습니다.

```
브라우저 (https://drspike.synology.me:4443)
  |
  | (1) DDNS로 공유기 공인 IP 확인
  v
공유기 (외부 4443 -> NAS 내부 IP:4443 포트포워딩)
  |
  | (2) NAS가 4443 포트로 HTTPS 요청 수신
  v
DSM 역방향 프록시 (HTTPS 4443 -> http://localhost:3000)
  |
  | (3) Docker 컨테이너의 매핑된 3000번 포트로 전달
  v
Docker: closingshin 컨테이너 (Port 3000 = Next.js)
```

### 4-1. Docker 측 포트 설정 (docker-compose.yml)

Docker 컨테이너 자체는 **HTTPS를 모릅니다**. 컨테이너는 순수하게 HTTP(3000, 3001)만 열어두면 됩니다. SSL 암호화는 시놀로지 DSM의 역방향 프록시가 대신 처리합니다.

```yaml
# docker-compose.yml 포트 설정 (변경 불필요)
services:
  closingshin:
    ports:
      - "3000:3000"   # Next.js 웹 서버 (역방향 프록시가 여기로 전달)
      - "3001:3001"   # Scheduler 상태 API
```

> [!IMPORTANT]
> Docker 컨테이너의 포트 설정에 4443을 넣으면 안 됩니다.
> - 4443은 DSM(시놀로지 운영체제)이 직접 수신하는 포트입니다.
> - Docker 컨테이너를 거치는 것이 아니라, DSM 역방향 프록시가 먼저 4443을 받아서 해독(SSL Termination)한 뒤 HTTP로 변환하여 Docker의 3000번 포트로 전달하는 구조입니다.

### 4-2. DSM 역방향 프록시 설정

**DSM 제어판 > 로그인 포털 > 고급 > 역방향 프록시** 에서 규칙을 생성합니다.

| 항목 | 출처 (외부) | 대상 (내부) |
|---|---|---|
| **프로토콜** | HTTPS | HTTP |
| **호스트명** | `drspike.synology.me` | `localhost` |
| **포트** | `4443` | `3000` |

이 설정의 의미: "외부에서 HTTPS 4443 포트로 들어온 요청을 SSL 해독한 뒤, NAS 내부의 Docker 컨테이너가 열고 있는 HTTP 3000번 포트로 넘겨라."

### 4-3. 공유기 포트포워딩 설정

공유기 관리자 페이지에서 다음 포트를 NAS의 내부 IP로 포워딩합니다.

| 외부 포트 | 내부 포트 | 프로토콜 | 대상 IP |
|---|---|---|---|
| `4443` | `4443` | TCP | NAS 내부 IP (예: 192.168.x.x) |
| `80` | `80` | TCP | NAS 내부 IP (Let's Encrypt 인증서 갱신용) |

> [!NOTE]
> 443 대신 4443을 사용하는 이유: 일부 공유기는 443 포트를 자체 관리 페이지(Remote Management)에 사용하므로 충돌이 발생할 수 있습니다. 4443처럼 대체 포트를 쓰면 이 문제를 깔끔하게 피할 수 있습니다.

### 4-4. SSL 인증서 설정

**DSM 제어판 > 보안 > 인증서** 에서 Let's Encrypt 인증서를 발급받습니다.
- 도메인: `drspike.synology.me`
- 발급 후, **인증서 > 설정** 에서 방금 만든 역방향 프록시 규칙에 해당 인증서를 매핑합니다.

### 요약: 누가 무엇을 담당하는가

| 담당 | 역할 | 포트 |
|---|---|---|
| **공유기** | 외부 트래픽을 NAS로 전달 | 4443 -> NAS:4443 |
| **DSM 역방향 프록시** | SSL 암호화/해독 + HTTP 변환 | HTTPS:4443 -> HTTP:3000 |
| **Docker 컨테이너** | 순수 HTTP 앱 서빙 | 3000 (Next.js), 3001 (Scheduler) |

## 확인 방법

빌드와 실행이 모두 끝난 후, 원래 접속하시던 URL 주소로 대시보드에 들어갑니다.
**Update Now** 버튼을 눌렀을 때, 화면이 새로고침되면서 과거 시점이 아닌 **최신 시간의 데이터가 즉각적으로 화면에 반영**된다면 캐싱 우회(force-dynamic) 업데이트가 완벽하게 적용된 것입니다!
