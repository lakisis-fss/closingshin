# ClosingSHIN 개발 환경 실행 및 관리 절차

현재 ClosingSHIN 프로젝트는 PocketBase(DB), Python Scripts(데이터 수집), Next.js(프론트엔드) 세 부분으로 구성되어 있습니다. 개발 단계에서 앱을 실행하고 관리하기 위한 절차를 제안합니다.

## 맥락
이전의 로컬 파일 기반 저장 방식에서 PocketBase 통합 환경으로 전환되었습니다. 실시간 시장 데이터와 VCP 스캔 결과를 화면에 표시하기 위해서는 DB 서비스, 데이터 수집기, 웹 서버가 유기적으로 실행되어야 합니다.

---

## 선택지 A: 완전 로컬 수동 관리
개발자의 PC에서 모든 서비스를 직접 관리하는 방식입니다.

- 1단계: PocketBase 실행: `pocketbase.exe serve` 명령어로 DB 활성화 (`http://127.0.0.1:8090`)
- 2단계: 데이터 수집: `venv` 환경에서 `05_collect_market_status.py` 등을 실행하여 DB에 데이터 주입
- 3단계: 프론트엔드 실행: `frontend` 폴더에서 `npm run dev` 실행

장점:
- 외부 서버 환경에 의존하지 않고 오프라인에서도 모든 기능 테스트 가능
- 로그를 즉시 확인하고 코드 수정을 빠르게 반영할 수 있음

단점:
- 매번 수동으로 수집 스크립트를 돌려야 최신 데이터가 유지됨
- PC 리소스를 많이 점유함 (특히 VCP 전수 조사의 경우)

노력 수준: 보통

---

## 선택지 B: 하이브리드 자동화
데이터 수집과 DB는 상시 구동되는 서버(예: Synology Docker 또는 Vercel/GitHub Actions)에 맡기고, UI 개발만 로컬에서 진행하는 방식입니다.

- 1단계: DB 연결: `.env` 파일의 `PB_URL`을 외부 서버 주소로 설정
- 2단계: 스케줄러 활용: 외부 서버에서 정기적으로 수집 스크립트 상시 구동
- 3단계: 로컬 UI 작업: 개발자는 `npm run dev`만 사용하여 실시간으로 쌓이는 데이터를 확인하며 작업

장점:
- 개발 시작 시 매번 데이터를 직접 수집할 필요가 없음
- 실제 서비스 환경과 가장 유사한 데이터 흐름을 확인 가능

단점:
- 외부 서버의 가용성에 따라 로컬 앱이 영향을 받음
- 네트워크 지연이 발생할 수 있음

노력 수준: 낮음 (초기 설정 후)

---

## 선택지 C: 대시보드 내 통합 관리
프론트엔드 내에 구현된 시스템 API를 사용하여 브라우저에서 제어하는 방식입니다.

- 1단계: PocketBase와 Next.js 서버만 기동
- 2단계: 대시보드 상단의 "Update Now" 또는 "Run Scan" 버튼 클릭
- 3단계: 프론트엔드 API가 백그라운드 프로세스로 Python 스크립트를 호출하여 데이터 갱신

장점:
- 명령줄을 만질 필요 없이 UI 상에서 데이터 흐름 제어 가능
- 비개발자도 데이터 수집 현황을 쉽게 관리할 수 있음

단점:
- 웹 서버가 직접 스크립트를 실행하므로 보안 설정과 리소스 관리가 중요함
- 스캔이 진행되는 동안 웹 서버 성능에 일시적인 부하가 올 수 있음

노력 수준: 높음 (초기 구현 및 안정화 필요)

---

## 권장 사항
선택지 C (대시보드 통합 관리)를 권장합니다.
이미 시스템 API와 Python 스크립트 연동을 마쳤으므로, 앱을 실행한 후 대시보드 내 버튼으로 시장 상황을 업데이트하는 현재의 방식이 가장 직관적입니다.

### 실행 순서 요약
1. PocketBase 서버 실행
2. Next.js 서버 실행 (`npm run dev`)
3. 브라우저 접속 후 "Update Now" 클릭 (시장 데이터 수집)
4. 필요 시 "Run Scan" 클릭 (VCP 스캔 시작)
