# Docker 환경에서 'Update Now' 캐싱 현상 해결 문서

## 문제 상황
- **증상:** 로컬 개발 환경에서는 'Update Now' 버튼을 누르면 실시간으로 데이터를 잘 가져옵니다. 그러나 시놀로지 도커 환경(실제 운영 배포판)에서는 버튼을 눌러 파이썬 데이터 크롤링 스크립트가 돌아가더라도 화면 상에는 아무런 변화가 없고, 2월 18일 시점의 내용만 고정적으로 보였습니다.
- **예시 설명:** 마치 식당(Next.js 서버)에서 "신선한 요리(실시간 데이터)"를 주문했는데, 주방장(개발 환경)일 때는 주문이 들어올 때마다 새로 요리해서 주다가, 공장(프로덕션 도커 빌드)이 되니까 아예 "2월 18일에 만든 밀키트 300인분"을 찍어내버리고 손님이 주문할 때마다 그 밀키트만 계속 전자레인지에 데워주는 것과 똑같은 상황입니다. 

## 원인: Next.js의 정적 캐시(Static Data Fetching) 전략
- Next.js는 사이트 접속 속도를 극단적으로 올리기 위해서, 빌드할 때(도커 이미지를 구울 때) API나 페이지들을 미리 한 번 싹 다 실행해버립니다. 그리고 그때 가져왔던 결과값을 메모리나 파일로 꽉꽉 얼려버립니다(베이킹, Baking).
- 앱이 로컬 컴퓨터의 파일(`Scripts/data` 폴더 안의 `.json` 파일)을 주기적으로 업데이트하고 이걸 화면에 실시간으로 뿌려주는 방식인데, Next.js 프로덕션 서버는 "어? 나 빌드할 때 2월 18일꺼 읽어서 얼려놨는데 굳이 또 읽어야 돼?" 라고 생각하고 새로 파일을 읽지 않습니다.
- 결국 파이썬 스크립트가 백그라운드에서 백날 새로운 데이터를 `.json`에 써봐야 헛수고였던 것입니다. 

## 해결 방법: 동적 렌더링(Dynamic Rendering) 강제화
이 앱처럼 시스템 디렉토리의 파일이 실시간으로 변하고 그걸 즉시 화면에 뿌려줘야 하는 경우는 Next.js의 똑똑한(?) 밀키트 기능을 완전히 꺼버려야 합니다. 이를 '동적 렌더링 강제화(force-dynamic)'라고 부릅니다.

1. **페이지 전체 강제화:** `/frontend/src/app/layout.tsx` (전체 화면의 뼈대) 최상단에 `export const dynamic = 'force-dynamic';` 이라는 한 줄을 넣어서 "이 화면들은 절대로 캐시에 얼려두지 마!" 라고 선언했습니다.
2. **모든 API 라우트 강제화:** `.json` 파일들(`data`, `results`)을 가져와서 뿌려주는 `/api/market-status/route.ts` 등 17개의 파일 최상단에도 똑같이 `force-dynamic` 문구를 일괄 추가하여, 누군가 데이터를 달라고 할 때마다 항상 최신 디스크 상태를 읽도록 만들었습니다.

이 조치로 인해, 다시 도커 빌드를 하시면 이제는 'Update Now' 버튼 동작 시 캐시된 내용 대신 싱싱한 최신 데이터를 다시 화면에 뿌려주게 됩니다!
