# 포트폴리오 청산조건 시뮬레이션 (Portfolio Exit Simulation Logic)

## 📌 문제 정의 (Problem Definition)
현재 `exit_monitor.py`와 `07_calc_portfolio.py`의 로직은 **오직 가장 최근의 종가(Close Price)** 만을 기준으로 손절/익절(Stop Loss / Take Profit) 조건을 검사합니다.
이로 인해 다음과 같은 치명적인 오류가 발생하고 있습니다.
- **과거 진입 청산 누락:** 예를 들어, 경인양행을 2026-01-28 4835원에 진입했다고 등록했을 때, 2026-02-05 고가 5400원(약 +11.6% ~ +15% 수익 구간)에 도달하여 이미 익절 조건이 충족되었음에도, 오늘 기준의 종가가 그보다 낮으면 시스템은 청산되지 않은 것으로(OPEN) 잘못 계산합니다.
- **장중 꼬리 터치 무시:** 당일 장중에 고가나 저가를 터치하여 청산 조건에 도달했으나, 종가가 다시 회귀한 경우에도 청산을 놓치게 됩니다.

---

## 💡 해결 방안: 일별 고점/저점 순차적 시뮬레이션 

포트폴리오에 진입일(`buyDate`)과 청산 조건(Exit Conditions)이 설정되어 있다면, 시스템은 진입일로부터 현재까지 매일의 **고가(High)** 와 **저가(Low)** 를 순차적으로 검사하는 백테스팅(시뮬레이션)을 반드시 거쳐야 합니다.

### 1단계: 진입 시점부터의 일봉 데이터 확보
해당 종목이 `OPEN` 상태라면 `FinanceDataReader` 등을 통해 진입일(`buyDate`)부터 오늘까지의 일봉 데이터(OHLCV)를 가져옵니다.

### 2단계: 일별 고가/저가 순차적 검사 (Daily OHLC Loop)
가져온 일봉 데이터를 과거부터 오늘까지 하루씩 순회하면서 청산 목표가 도달 여부를 확인합니다.

1. **익절 (Take Profit) 검사**
   - **조건식**: `당일 High >= 목표 익절가` (예: 진입가 + 15%)
   - 당일 고가(High)가 목표가에 도달하거나 더 높다면, 그 날짜에 익절 청산된 것으로 처리합니다.

2. **손절 (Stop Loss) 검사**
   - **조건식**: `당일 Low <= 목표 손절가` (예: 진입가 - 7%)
   - 당일 저가(Low)가 손절가에 도달하거나 더 낮다면, 그 날짜에 손절 청산된 것으로 처리합니다.

> [!NOTE] 엣지 케이스 (Edge Case) 우선순위 해결
> 만약 캔들의 범위가 커서 같은 날 고가와 저가가 모두 익절가와 손절가를 터치하는 경우, 사용자의 요청에 따라 **익절(Take Profit)이 먼저 적용되었다고 간주**합니다. 즉, 동일 일자에 양 조건을 모두 충족했다면 시스템은 성공적인 익절 청산으로 기록합니다.

### 3단계: 포트폴리오 상태 영구 업데이트 (`portfolio.json`)
조건이 한 번이라도 충족되었다면 해당 날짜 즉시 순회를 멈추고 시스템의 `portfolio.json` 상에서 상태를 영구적으로 `CLOSED`로 업데이트합니다.
- `status`: 'CLOSED' (기존 'OPEN'에서 변경)
- `exitDate`: 조건 충족된 정확한 날짜
- `exitPrice`: 익절가 또는 손절가 (실제 체결된 가격으로 간주)

## 🛠️ 수정이 필요한 스크립트

이를 해결하기 위해 다음의 파이썬 파일들에 변경이 필요합니다.

1. **`Scripts/07_calc_portfolio.py`**
   - 가치평가를 수행하기 전에, 우선 `portfolio.json` 내의 OPEN 종목 중 과거 혹은 오늘 장중 청산 목표치에 도달한 것이 있는지 위 시뮬레이션 로직을 돌려 `CLOSED` 처리를 먼저 완료합니다. 해당 변동사항을 실제 `portfolio.json` 원본 파일에도 덮어쓰기하여 상태를 동기화합니다.
2. **`Scripts/exit_monitor.py`**
   - `07_calc_portfolio.py`가 과거 정산을 마쳤다는 전제하에, 알림 트리거 등에 오작동이 없도록 로직을 일치시키거나, 모니터 스크립트 역시 고가/저가를 기준으로 판별하도록 논리를 정리합니다.
