# Brainstorm: 스케줄러 작동 시간 및 구조 개선 방안

## Context
현재 프로젝트에는 두 개의 스케줄러가 개별적으로 돌아가고 있습니다. 스케줄러가 언제 어떻게 동작하는지 명확히 정리하고, 향후 관리의 편리함을 위해 어떤 방향으로 개선할 수 있을지 브레인스토밍합니다.

---

## 1. 현재 스케줄러가 작동하는 시간

프로젝트 안에는 크게 두 명의 스케줄러가 존재합니다. 하나는 전체 시스템 데이터를 주기적으로 수집하는 역할(Node.js)이고, 다른 하나는 주식 장이 열려 있을 때만 집중 감시하는 역할(Python)입니다.

### 첫 번째: Node.js 메인 스케줄러 (frontend/backend/scheduler.ts)
이 스케줄러는 24시간 내내 켜져서 전반적인 데이터를 가져오고 스캔 결과를 분석합니다.
- **Market Data Update (시장 데이터 업데이트):** 매 **10분마다** 반복해서 시장의 상태를 확인합니다.
- **VCP Scan (VCP 패턴 종목 스캔):** 매일 **오후 3시 40분 (KST)**에 딱 한 번 실행됩니다. 주식 장이 3시 30분에 끝나기 때문에, 끝난 직후 결과를 모아서 분석하는 것입니다.

### 두 번째: Python 장중 감시 스케줄러 (Scripts/scheduler.py)
이 스케줄러는 주식 장이 열려 있는 요일과 시간에만 집중적으로 동작합니다.
- **Market Hour Monitor (장중 엑시트 모니터링):** **월요일부터 금요일까지, 오전 9시부터 오후 3시 50분 사이**에 매 **10분마다** 실행됩니다. 내가 보유한 주식의 가격이 떨어지면 팔아야 하는지(Exit) 감시합니다.
- **Post-Market Check (장 마감 점검):** **월요일부터 금요일까지, 오후 6시 정각**에 딱 한 번 실행되어 장 마감 후 최종 상태를 확인합니다.

---

## 2. 향후 스케줄러 구조 개선 옵션 (Options)

현재 두 개로 스케줄러가 나뉘어 있다 보니, 나중에 작동 일정을 바꿀 때 두 군데를 모두 확인해야 합니다. 이를 개선하기 위한 3가지 방법입니다.

### Option A: 하나의 스케줄러로 전면 통합하기
Node.js 스케줄러가 Python 프로그램들의 일정까지 전부 직접 실행하도록 통일하는 방법입니다.

- **Pros:**
  - 코드를 관리하기 매우 편해집니다. 일정 변경 시 파일 하나만 수정하면 됩니다.
  - 일정이 겹치는 문제를 방지할 수 있습니다.
- **Cons:**
  - Node.js 스케줄러가 꺼지면 전체 작업이 즉시 멈춥니다.
- **Effort (작업량):** Low (낮음)

### Option B: Redis (메시지 큐)를 도입해 유기적으로 연결하기
스케줄러가 직접 실행 명령을 내리는 대신, 명령 전달 게시판(Redis)에 수행할 작업을 올리면 개별 실행기들이 이를 받아서 일하는 구조입니다.

- **Pros:**
  - 각 스케줄러가 파트별로 분리되어 독립적으로 일하므로 안정적입니다.
  - 서버를 추가하거나 규모를 키우는데 적합합니다.
- **Cons:**
  - Redis라는 새로운 시스템을 배우고, 인프라를 설치하는 복잡한 과정이 추가됩니다.
- **Effort (작업량):** High (높음)

### Option C: 크론탭(OS 자체 스케줄러)에 맡기기
프로젝트 코드 안에서 무한 반복 타이머를 돌리지 않고, 리눅스나 윈도우 운영체제가 제공하는 '예약 작업'(Crontab / Task Scheduler)에 실행 권한을 맡기는 방법입니다.

- **Pros:**
  - 프로그램 안에서 타이머가 무한정 도는 불필요한 공회전을 줄여서 서버 자원을 아낄 수 있습니다.
- **Cons:**
  - 코드가 아닌 운영체제 설정에 접근해야 하므로 터미널 사용법이나 권한 관리에 익숙해져야 합니다.
- **Effort (작업량):** Medium (보통)

---

## Recommendation (추천 방안)

**Option A (하나의 스케줄러로 전면 통합하기)** 를 추천합니다.
`scheduler.ts` 파일 한 곳에서 모든 시간표를 파악할 수 있기 때문입니다. 소규모 및 개인 프로젝트 구조에서는 '관리의 간편함'이 가장 큰 장점이므로 여러 곳을 오가며 설정하는 불편함을 줄이는 것을 추천합니다.
