# 시스템 로그 삭제 무한 루프 문제 해결 보고서

## 1. 문제 증상
- 관리자 페이지의 시스템 로그 컴포넌트에서 **휴지통(삭제) 버튼**을 클릭했을 때, 로그가 삭제된 후 화면에 "Waiting for logs..." 메시지와 로딩 스피너가 사라지지 않고 계속 표시됨.
- 사용자는 이를 시스템의 '무한 루프'나 '멈춤 현상'으로 인식하게 됨.

## 2. 발생 원인 분석
이 문제는 크게 두 가지 원인이 복합적으로 작용하여 발생했습니다.

### A. Windows 환경에서의 파일 잠금 (Backend)
- 기존 백엔드 코드는 로그 삭제 시 `fs.unlinkSync`를 사용하여 파일을 아예 삭제하려고 시도했습니다.
- 하지만 현재 시스템은 `npm run scheduler`와 같은 프로세스가 로그 파일(`debug_output.txt`)에 계속해서 기록을 남기고 있는 상태입니다.
- **Windows 운영체제**는 다른 프로세스가 사용 중인 파일을 삭제(unlink)하는 것을 기본적으로 차단합니다. 이로 인해 삭제 요청이 실패하고 서버 에러(500)가 발생할 가능성이 높았습니다.

### B. UI 상태 관리 미흡 (Frontend)
- 프론트엔드 코드에서 로그 배열의 길이가 0(`logs.length === 0`)일 때, 실제 로딩 중인지 아니면 데이터가 없는 것인지 구분하지 않고 무조건 로딩 스피너와 "Waiting for logs..." 메시지를 표시하도록 설계되어 있었습니다.
- 설령 삭제가 성공했더라도 로그가 비어있기 때문에 사용자는 영원히 로딩 중인 화면을 보게 되었던 것입니다.

## 3. 해결 대책

### A. 백엔드: 안전한 로그 비우기 방식 도입
- 파일을 삭제하는 대신, **파일의 내용을 비우는(Truncate)** 방식(`fs.writeFileSync(LOG_FILE, '')`)으로 변경했습니다.
- 이 방식은 파일 자체를 지우지 않으므로, 다른 프로세스가 파일을 열고 있어도 비교적 안전하게 내용을 초기화할 수 있습니다.

### B. 프론트엔드: 사용자 경험(UX) 개선
- **상태 구분**: 로딩 중(`loading === true`)일 때만 스피너를 보여주고, 로딩이 끝났는데 데이터가 없다면 "로그 기록이 없습니다."라는 안내 문구와 터미널 아이콘을 표시하도록 수정했습니다.
- **에러 핸들링**: 삭제 요청이 실패할 경우 사용자에게 알림창(alert)을 띄워 상황을 인지할 수 있게 했습니다.
- **버튼 제어**: 로딩 중에는 중복 클릭을 방지하기 위해 버튼을 비활성화했습니다.

## 4. 결과
이제 로그를 삭제하면 즉시 "로그 기록이 없습니다."라는 안내가 표시되며, 로딩 스피너가 무한히 도는 현상이 해결되었습니다. 새 로그가 생성되면 '새로고침' 버튼을 통해 다시 확인할 수 있습니다.
